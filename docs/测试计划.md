## 一、概述

### 1.1 测试目标

确保 Dora 魔盒 SaaS 产品从灵感输入到视频发布的全流程功能正常、性能达标、用户体验流畅。通过系统化的测试活动，保障产品质量，降低上线风险。

### 1.2 测试范围

- **功能测试**：剧本生成、分镜拆解、画面绘制、视频生成、视频拼接、视频下载、视频发布
- **非功能测试**：性能、安全性、兼容性、可靠性
- **测试层级**：单元测试、集成测试、端到端测试

### 1.3 质量标准

| 指标 | 目标值 |
|------|--------|
| 单元测试覆盖率 | ≥ 80% |
| 核心功能测试通过率 | 100% |
| 接口响应时间 (P95) | < 2s |
| 系统可用性 | ≥ 99.5% |

## 二、单元测试规范

### 2.1 测试框架

| 层级 | 框架 | 说明 |
|------|------|------|
| Go 单元测试 | testing 标准库 | 配合 testify/assert |
| HTTP API 测试 | hertz 的测试工具 | 模拟 HTTP 请求 |
| RPC 服务测试 | kitex 的测试工具 | 模拟 RPC 调用 |

### 2.2 目录结构

```
dora-magic-box/
├── internal/
│   ├── api/
│   │   └── **/*_test.go          # API 层单元测试
│   ├── biz/
│   │   └── **/*_test.go          # 业务逻辑层单元测试
│   ├── dal/
│   │   └── **/*_test.go          # 数据访问层单元测试
│   └── pkg/
│       └── **/*_test.go          # 公共包单元测试
└── test/
    └── utils/                     # 测试工具和辅助函数
```

### 2.3 命名规范

- **测试文件**：`{被测试文件名}_test.go`
- **测试函数**：`Test{函数名/结构体名}_{场景}`
- **表格测试**：`Test{函数名/结构体名}_TableDriven`
- **示例**：
  ```go
  // 普通测试
  func TestGenerateScript_Success(t *testing.T) {}
  func TestGenerateScript_InvalidInput(t *testing.T) {}

  // 表格驱动测试
  func TestValidatePrompt_TableDriven(t *testing.T) {}
  ```

### 2.4 测试结构

每个测试函数应遵循 AAA 模式：

```go
func TestXxx(t *testing.T) {
    // Arrange - 准备测试数据和环境
    input := "..."
    expected := "..."

    // Act - 执行被测试的代码
    result := FunctionUnderTest(input)

    // Assert - 验证结果
    assert.Equal(t, expected, result)
}
```

### 2.5 Mock 策略

| 依赖 | Mock 方式 | 说明 |
|------|-----------|------|
| 数据库 | GORM 内存数据库 / sqlmock | 使用 SQLite 或 go-sqlmock |
| Redis | miniredis | 轻量级 Redis Mock |
| RPC 服务 | gomock | Kitex 服务接口 Mock |
| 外部 API | httptest | HTTP 测试服务器 |

### 2.6 测试覆盖率要求

| 包类型 | 覆盖率目标 | 说明 |
|--------|-----------|------|
| 业务逻辑层 (biz) | ≥ 85% | 核心业务逻辑必须覆盖 |
| 数据访问层 (dal) | ≥ 75% | 包含事务、错误处理 |
| 工具包 (pkg) | ≥ 90% | 通用工具需高覆盖率 |
| API 层 | ≥ 70% | 主要测试参数校验 |

### 2.7 单元测试检查清单

- [ ] 正常流程测试
- [ ] 边界条件测试（空值、最大值、最小值）
- [ ] 异常路径测试（错误输入、依赖失败）
- [ ] 并发安全测试（如适用）
- [ ] 测试数据清理（每个测试独立）

| 性能测试 | 迭代末期 | 测试/架构师 | 性能指标验证 |
| 回归测试 | 每次发布前 | 测试人员 | 全量回归 |

### 6.2 缺陷管理

| 严重等级 | 定义 | 响应时间 | 修复时限 |
|---------|------|---------|---------|
| P0 - 致命 | 系统崩溃、数据丢失、核心功能不可用 | 立即 | 4 小时 |
| P1 - 严重 | 主要功能不可用、性能严重下降 | 2 小时 | 24 小时 |
| P2 - 一般 | 次要功能问题、交互问题 | 4 小时 | 3 天 |
| P3 - 轻微 | UI 问题、文案问题、建议 | 1 个工作日 | 1 周 |
